TasteKit

Human operating taste -> portable agent runtime artifacts

TasteKit captures how a person wants agents to reason, challenge, verify, communicate, remember, bind tools, and adapt. The output is not a prompt blob: it is a versioned system of artifacts, runtime exports, traces, evals, trust checks, and reviewed drift proposals.

Six-Domain Grid

Domains are not labels only. Each domain brings rubrics, skills, playbooks, guardrails, fixtures, and release checks.

general-agent

General Agent

Job: broad planning, synthesis, research-then-act, and decision support.

Specializes: mixed rubric, context synthesis, task orchestration.

Focus: evidence rigor, risk escalation, communication contract.

development-agent

Development Agent

Job: code review, debugging, testing, refactoring, and engineering docs.

Specializes: code quality, test philosophy, type safety, CI practice.

Focus: review depth, change scope, and verification style.

content-agent

Content Agent

Job: editorial drafting, adaptation, brand voice, and publishing boundaries.

Specializes: voice briefs, claim policy, channel strategy, workflow.

Focus: anti-generic standards and approval before publish.

research-agent

Research Agent

Job: source discovery, evidence grading, synthesis, and competitive analysis.

Specializes: source quality, citation expectations, uncertainty handling.

Focus: weak-source escalation and explicit caveats.

sales-agent

Sales Agent

Job: account research, qualification, follow-up, and deal-risk escalation.

Specializes: buyer relevance, CRM boundaries, claim discipline.

Focus: approval points for external commitments.

support-agent

Support Agent

Job: troubleshoot issues while preserving privacy, accuracy, and escalation.

Specializes: triage, customer response drafting, and handoff clarity.

Focus: security, billing, and account-data guardrails.

Canonical versus generated

The constitution keeps stable schema fields while preserving depth in extension keys. Generated runtime files are practical projections from that source, not the source of truth.

Artifact-first behavior

Everything compiles to files. Adapters map those files into the target runtime surface while preserving user-managed text outside TasteKit regions.

Trace-first maintenance

Agent runs can create traces that feed eval, replay, drift detection, and trust review instead of relying on memory of what happened.

Schema-first contracts

Versioned JSON and YAML artifacts keep release validation grounded in typed contracts rather than prose-only expectations.

Posture

Trust, Safety, And Privacy

This is a conservative runtime posture: inspect before binding, pin provenance, validate schemas, and keep hidden interview mechanics out of runtime Markdown.

MCP-first

Tool discovery and binding use Model Context Protocol rather than a custom TasteKit protocol.

no auto-enable

New or changed tools should be inspected before they affect runtime behavior.

trust pins

MCP servers and skill sources are pinned for provenance and audit visibility.

Validator

Schema checks and contract gates keep artifacts reviewable before release or runtime handoff.

Privacy boundary

Runtime files summarize guidance and exclude raw transcripts or hidden policy machinery.

Behavior over time

Maintenance Loop

Maintenance is a core surface, not a later add-on. Drift detection proposes reviewable updates; a human decides what changes.

Run -> Trace -> Eval -> Drift Detect -> Human Review -> Update Taste

Traces provide machine-readable evidence. Evals and drift checks can identify mismatches between observed behavior and the profile, but updates remain proposals until reviewed.

This keeps taste maintenance visible: the product can detect stale or mismatched behavior without silently rewriting a person's operating profile.

Review boundaries

Risks, Gaps, And Unknowns

These labels keep the one-pager honest. Some surfaces are release blockers, and some are explicitly Deferred rather than shipped.

Release gates still decide publication

The project remains release-readiness work until deterministic gates, live release evidence, package checks, and human review pass.

Provider availability matters

Live release evidence depends on accepted official endpoints, model availability, and the strict GPT-5.5 plus GLM-5.1 route.

Public claims must stay conservative

Docs should distinguish current implementation, release-readiness status, subscription review evidence, and official release path.

Deferred

Hosted dashboard, team workflows, native runtime control beyond exports, clinical or diagnostic claims, fully automatic drift application, and skill auto-update are outside the shipped claim.
